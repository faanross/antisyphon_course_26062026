1 Download the repo

Pick whichever is easier — both end with the project folder on your machine.

Where to put it

Clone or unzip into a folder you own — your home directory or Desktop is ideal. Do not put it under a system path like /opt, /usr, or C:\Program Files, and don't clone with sudo — the labs need to write files (a Python environment for Lab 05), and a system-owned folder blocks that.

Option A — Git (recommended)

If you have git installed, clone it:

$ git clone https://github.com/faanross/antisyphon_course_26062026.git
$ cd antisyphon_course_26062026

Option B — Download a ZIP (no git needed)

1

Open github.com/faanross/antisyphon_course_26062026 in a browser.

2

Click the green Code button → Download ZIP.

3

Unzip it somewhere easy to find that you own (e.g. your Desktop or home folder), then cd into the unzipped folder in your terminal.

2 Run the installer script

From the project folder, move into hunting-agent/ and run the script for your OS. It is safe to re-run.

macOS / Linux

$ cd hunting-agent
$ bash ./setup.sh

Windows (PowerShell)

> cd hunting-agent
> powershell -ExecutionPolicy Bypass -File setup.ps1

What the script actually does

  • Checks whether you already have a supported Node (20.19+, 22.12+, or 24+; it deliberately avoids 21/23).
  • If not, it installs Volta (a small, cross-platform Node version manager) and pins Node 22.12.0 — without touching any other Node you might have.
  • Runs npm install to download the app's dependencies.
Windows note

The -ExecutionPolicy Bypass part is required — without it PowerShell refuses to run scripts. On Windows the script uses winget to install Volta; if you don't have winget, install Volta manually from volta.sh, reopen PowerShell, and re-run.

If Node was just installed

Close the terminal, open a fresh one, cd back into hunting-agent, and run the script again. Volta updates your PATH, and a new terminal is the cleanest way to pick it up.

3 Choose how to run the model

Every lab calls a real language model. You have three ways to provide one — you only need one. Pick the path that matches what you already have, then jump to its part below.

Path A · A CLI you already pay for

Already subscribe to Claude (Claude Code) or ChatGPT (Codex CLI) and have the CLI installed? Use it directly — no API key, no extra cost beyond your subscription. → Part A

Path B · A local model with Ollama

Free, private, runs entirely on your own machine (offline). Needs a reasonably capable computer. → Part B

Path C · An API key

Gemini, OpenAI, or Anthropic. Gemini has a free tier that's perfect to get started. → Part C

Not sure which?

If you already have a Claude or ChatGPT subscription, Path A is the least setup and costs nothing extra — you do not need to buy an API key. If you don't, a free Gemini key (Path C) is the fastest way to get going. Path B is for those who'd rather run everything locally.

First, create your .env file (all paths need this)

The app reads your choice from a file called .env. Create it from the template:

# macOS / Linux
$ cp .env.example .env

# Windows (PowerShell)
> copy .env.example .env
One provider at a time

Open .env in any text editor. Set the single LLM_PROVIDER line to your choice, fill in only that path's values, and make sure every line belonging to the other paths is commented out with a leading #. Mixing two active providers is the most common cause of confusing errors.

A Use a CLI you already subscribe to

If you already pay for Claude or ChatGPT and have its command-line tool installed and signed in, the app drives that tool for you. There is no API key in .env, and you pay nothing beyond the subscription you already have. This is the simplest path for existing subscribers.

A1 · Claude Code (Anthropic subscription)

1

Install the CLI (needs Node, which Section 2 set up): npm install -g @anthropic-ai/claude-code

2

Sign in: run claude, then type /login and complete the browser login with your Claude account.

3

In .env set just these (comment out everything else):

LLM_PROVIDER=claude-code
CLAUDE_CODE_MODEL=sonnet
# CLAUDE_CODE_MODEL=opus   # if your plan includes it
If you later see "exited with code 1"

That's usually a one-time auth hiccup when the tool is run non-interactively. Run claude setup-token to mint a token for programmatic use, then retry. Full steps in Section 8 → Fix CC-1.

Anthropic announced that from June 15 2026, claude -p / Agent SDK usage would move off your normal plan limits onto a separate monthly credit (one-time opt-in; Pro $20 / Max 5x $100 / Max 20x $200, metered at API rates). As of writing the rollout is unsettled — there are reports it may be paused. So: nothing to do up front. If claude-code works normally, you're on the existing behavior and can ignore this. If it suddenly stops or mentions a "credit," see Fix CC-3. (If the change does go live, Free-plan accounts can't run claude-code — use Part B local or Part C API key.)

A2 · Codex CLI (OpenAI / ChatGPT subscription)

1

Install the Codex CLI (needs Node 22+, which Section 2 set up): npm install -g @openai/codex

2

Sign in: run codex and choose Sign in with ChatGPT (a browser opens). Codex is included with ChatGPT Plus / Pro / Business / Edu / Enterprise plans — no separate purchase. On a headless box, use codex login --device-auth for a paste-a-code flow.

3

In .env set just these (comment out everything else):

LLM_PROVIDER=codex-cli
CODEX_MODEL=gpt-5.4
# CODEX_MODEL=gpt-5.4-mini
How to tell it's ready

Before touching the app, make sure the bare CLI works in your terminal: claude --print "hi" (Claude Code) or a quick codex run (Codex). If the CLI answers, the app will too. The app simply runs claude / codex for you, so they must be on your PATH and signed in.

B Run a local model with Ollama

Ollama (and LM Studio, llama.cpp, vLLM, LiteLLM) expose an OpenAI-compatible API. So you keep LLM_PROVIDER=openai and simply point the app at your local server with OPENAI_BASE_URL. No cloud, no key, no cost.

1

Install Ollama from ollama.com and make sure it's running (the desktop app runs the server automatically; or run ollama serve).

2

Pull a capable instruct model — 7B or larger. Tiny models mangle the structured-output labs:

$ ollama pull qwen2.5
3

In .env set these four lines (comment out everything else):

LLM_PROVIDER=openai
OPENAI_API_KEY=ollama                       # any non-empty string; local servers ignore it
OPENAI_MODEL=qwen2.5                         # whatever you `ollama pull`ed
OPENAI_BASE_URL=http://localhost:11434/v1   # Ollama's OpenAI-compatible endpoint

Running Ollama on another machine on your network (LAN)

If the model runs on a different box (e.g. a desktop with a GPU) and you want to use it from your laptop, point the base URL at that machine instead:

OPENAI_BASE_URL=http://192.168.1.50:11434/v1   # the GPU box's LAN IP
LAN requirement

By default Ollama only listens on localhost. On the machine running the model, start it bound to all interfaces — OLLAMA_HOST=0.0.0.0 ollama serve — and allow port 11434 through that machine's firewall.

Honest note

Local models are great for privacy and cost, but quality and speed vary with your hardware. The later structured-output labs (06 onward) need a capable instruct model; very small or "reasoning/<think>" models often produce malformed JSON. If a lab misbehaves on local, a free Gemini key (Path C) is a reliable fallback.

C Use an API key

Use a hosted model via an API key. Set LLM_PROVIDER to your choice, fill in that key/model, and comment out the rest.

LLM_PROVIDERWhat you needSet in .env
gemini An API key from Google AI Studio (has a free tier). GEMINI_API_KEY=…
GEMINI_MODEL=gemini-2.5-flash
openai An OpenAI API key (paid account). OPENAI_API_KEY=…
OPENAI_MODEL=gpt-5.4-mini
anthropic An Anthropic API key (paid account). ANTHROPIC_API_KEY=…
ANTHROPIC_MODEL=claude-haiku-4-5-20251001
Model IDs change

Model names get updated over time. If a model ID is rejected, swap it for a current one from the provider's model list — the links are in the comments at the top of .env.example.

Free vs paid

We're provider-agnostic — use whatever you have or prefer. Two honest notes: gemini-2.5-flash has a free tier, which is great for trying it, but the free tier is heavily rate-limited — during a full day of labs you'll likely hit 429/500s and interruptions. If you'd rather work uninterrupted, a paid API key (any provider) is smoother: the entire course costs roughly $5–10 of tokens at most, even with a stronger model such as Claude Sonnet.

Example: getting a free Gemini key (Google AI Studio)

1

Go to aistudio.google.com and sign in with a Google account.

2

In the left sidebar open Dashboard, then API Keys.

3

Click Create API key (top right). A key is created under a project on the Free tier.

4

Copy the key.

5

In .env set:

LLM_PROVIDER=gemini
GEMINI_API_KEY=your-key-here
GEMINI_MODEL=gemini-2.5-flash
# keep every other provider's lines commented out with #

Save the file. That's it — you're ready to run.

Keep it secret

Your .env holds a credential. Don't paste it into chat, screenshots, or commit it (the repo already ignores .env). Treat it like a password.

4 Set up the GTI lab (MCP)

Lab 05 makes a real call to Google Threat Intelligence over MCP — it's part of the course, not an extra. Set up its two prerequisites now, while you're still installing things, so the single fresh terminal in the next step picks up everything at once.

1. Install uv (a single binary; it fetches Python for you)

You can run this from anywhere — uv installs a global, system-wide binary; you do not need to be in hunting-agent/.

# macOS / Linux
$ curl -LsSf https://astral.sh/uv/install.sh | sh

# Windows (PowerShell) — run BOTH lines, in order
> Set-ExecutionPolicy RemoteSigned -Scope CurrentUser
> powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
Windows note

On a fresh Windows machine the execution policy is Restricted, and uv's installer refuses to run under it (you'll see "PowerShell requires an execution policy in [Unrestricted, RemoteSigned, Bypass]"). The first line fixes that — it only affects your user account and is the standard dev setting. Answer Y if it prompts, then run the install line. Don't restart anything yet — the next step opens one fresh terminal that picks up both Node and uv together.

2. Add a free VirusTotal API key to .env

Get one at virustotal.com/gui/my-apikey, then add this line to the .env you created in Section 3:

VT_APIKEY=your-virustotal-key
Expected

The MCP server is already bundled in the repo (mcp-security/) — no separate download. The first Lab 05 run may pause a few seconds while uv fetches the server's Python dependencies. That's normal, and only happens once. (If it errors instead, see Section 8 → Fix MCP-2.)

5 Start the server

Do this once — open ONE fresh terminal

Setup installed Node (via Volta) and uv, and both updated your PATH. Any terminal you already had open won't see those changes, so npm and uv will read as "not recognized." Close every PowerShell / terminal window — on Windows, fully quit Windows Terminal (the whole app, not just the tab) — then open one brand-new terminal and cd into hunting-agent/. Confirm both: npm -v and uv --version should each print a version. (If either is missing, sign out of Windows and back in.) Start the server from this terminal — the labs it launches inherit its PATH, which is how Lab 05 finds uv.

From inside hunting-agent/:

$ npm run dev

It prints a local URL — usually http://localhost:5173 (if that port is busy it picks the next one, so use whatever URL it prints). Open it in your browser. Leave this terminal running for the whole course; press Ctrl + C in it to stop the server.

Running in a VM or WSL?

If the app runs in a headless VM / WSL2 and you open it from a browser on the host, start it with npm run dev:host so it listens on all interfaces. See Section 8 → Fix NET-2.

6 Confirm it worked

A 30-second smoke test that proves the app, your model choice, and the wiring are all good:

1

On the landing page, open Lab 01 — Our First Agent.

2

Click the Agent tab, type anything (e.g. "What is beaconing?"), and press Send.

3

You should see a reply stream in word by word. That's success — the browser reached your local server, which called the model you configured. ✔

If it errors

An error here almost always means Section 3: provider not set, a key still left as paste-your-key-here, a model ID the provider rejected, or (Path A) a CLI that isn't signed in. Re-check Section 3, save .env, restart npm run dev — then use the guided Troubleshooting in Section 8.

8.2 · Fixes — Setup, Node & PATH

Fix SET-1 — npm/node "not recognized"

After a successful setup, the terminal can't find npm/node

Your terminal is using the old PATH. Close all terminals — on Windows fully quit Windows Terminal (the whole app, not just the tab) — open a brand-new one, cd back into hunting-agent/, and try again. Confirm with npm -v. Still missing? Sign out of Windows and back in (or reboot).

Fix SET-2 — node: command not found right after setup

Volta installed Node but this shell doesn't see it yet

Close the terminal, open a new one, cd back in, and re-run the setup script. Volta's PATH change needs a fresh shell to take effect.

Fix SET-3 — Windows "running scripts is disabled on this system"

PowerShell refused to run setup.ps1

You omitted the policy flag. Run the script with it exactly: powershell -ExecutionPolicy Bypass -File setup.ps1

Fix SET-4 — Windows "winget not found" during setup

The script couldn't use winget to install Volta

Install Volta manually from volta.sh (the MSI installer), reopen PowerShell, then re-run the setup script.

Fix SET-5 — npm install fails behind a corporate network

Dependency downloads time out or are blocked

You're likely behind a proxy or filtered network. Configure npm's proxy settings, or try again from a home / non-restricted network.

8.3 · Fixes — Model provider & .env

Fix ENV-1 — LLM_PROVIDER not set / "… not set in .env"

The app doesn't know which model to use

You didn't create/edit .env, or a required value is still paste-your-key-here. Re-do Section 3 for your chosen path, save the file, and restart the server.

Fix ENV-2 — Model errors / 401 / "invalid model"

The provider rejected your key or model ID

Wrong or expired key, or a model ID the provider no longer offers. Re-check the key, and pick a current model ID from the provider's list (links are in the comments at the top of .env.example). Save and restart.

Fix ENV-3 — App calls the wrong provider, and restarting doesn't help

e.g. a Google/Gemini error when .env clearly says openai

A stale exported environment variable (LLM_PROVIDER or a *_API_KEY) in your shell is overriding .env — the app's loader will not replace a variable that's already set, and it survives restarts. Check it:

# macOS / Linux
$ echo $LLM_PROVIDER
# Windows (PowerShell)
> $env:LLM_PROVIDER

If it prints anything, clear it — unset LLM_PROVIDER (PowerShell: Remove-Item Env:LLM_PROVIDER) — and remove any such lines from ~/.zshrc / your shell profile. Then restart the server.

8.4 · Fixes — Path A (CLI subscription)

Fix CC-1 — claude-code: "Claude Code exited with code 1"

The CLI ran but failed — usually a hidden 401 on a subscription login

When Claude Code is driven non-interactively it can reject a subscription session with a 401 even though the interactive app works. Fix the credential:

1

Run claude, type /login, and complete the browser login again.

2

If it still fails, run claude setup-token — this mints a token for non-interactive/programmatic use (the canonical fix for "interactive works, headless 401s").

3

Confirm with claude --print --model sonnet "hi"; once it answers, retry the lab.

Fix CC-2 — spawn claude ENOENT / spawn codex ENOENT

The CLI isn't on the PATH of the terminal running the server (may crash it → browser shows "NetworkError when attempting to fetch resource")

The app launches claude / codex by name, so it must be installed and on the PATH of the terminal that started npm run dev.

1

Stop the server. In a fresh terminal run claude --version (or codex --version).

2

Not recognized? Install it (Claude Code: npm install -g @anthropic-ai/claude-code; Codex: npm install -g @openai/codex), then close and reopen the terminal.

3

Sign in (claude → /login; or run codex → Sign in with ChatGPT). Then start npm run dev from that same terminal — the server inherits its PATH.

Fix CC-3 — claude-code stopped working after June 15 (Agent SDK credit)

If Anthropic's Agent SDK credit change is active for your account (announced for June 15 2026; rollout status was in flux), claude -p draws from a separate monthly credit you must claim — it no longer rides your normal plan limits

1

Be on a paid plan. The Free plan gets no Agent SDK credit, so claude-code won't run — use Path B (local) or Path C (API key) instead.

2

Claim the credit (one-time). Open your Claude account and opt in to the Agent SDK monthly credit (eligible users were emailed instructions). Until you claim it, headless claude -p may have no allowance to draw on.

3

Mind the cap: Pro $20/mo, Max 5x $100, Max 20x $200 — metered at API rates, reset monthly. Tool-heavy labs (esp. Lab 05) spend it. When it's gone, requests stop until next cycle — unless you've enabled paid usage credits, in which case they bill at standard API rates. To avoid watching a meter, switch .env to a local model (Path B) or an API key (Path C).

8.5 · Fixes — Path B (local / Ollama)

Fix LOC-1 — Can't reach the local model (connection refused)

ECONNREFUSED / connection refused / no response

1

Make sure the model server is actually running — open the Ollama app, or run ollama serve. Test it: curl http://localhost:11434/v1/models should return JSON.

2

Check your .env: LLM_PROVIDER=openai and OPENAI_BASE_URL points at the right host. Local: http://localhost:11434/v1. Another box: that machine's LAN IP.

3

Using a model on another machine? It must bind all interfaces (OLLAMA_HOST=0.0.0.0 ollama serve) and allow port 11434 through its firewall. Confirm the model name in OPENAI_MODEL matches one you ollama pulled.

Fix LOC-2 — model runner has unexpectedly stopped / garbled JSON

The model ran out of memory, or is too small for structured output

"Unexpectedly stopped" means the machine ran out of memory loading the model — use a smaller model, or run it on a stronger (GPU) box. For the structured-output labs (06 onward), use a capable instruct model (e.g. qwen2.5, llama3.1) — tiny or "reasoning/<think>" models often mangle the JSON. If local keeps misbehaving on those labs, a free Gemini key (Path C) is a reliable fallback.

8.6 · Fixes — Path C (API keys)

Fix API-1 — OpenAI "you exceeded your current quota" / insufficient_quota

Billing, not a lab bug

Your OpenAI account balance is $0. Add a payment method and a few dollars of credit (Settings → Billing). Credits take a minute or two to propagate; no .env change or restart needed.

Fix API-2 — Gemini free tier: frequent 429 / 500 interruptions

The free tier is heavily rate-limited

Expected on the free tier during a full day of labs. Either wait a moment and retry, or switch to a paid key (any provider) for smooth, uninterrupted runs — the whole course is roughly $5–10 of tokens. Update .env and restart.

8.7 · Fixes — Lab 05 (GTI / MCP)

Fix MCP-1 — -32000: Connection closed + 'uv' is not recognized

uv isn't on the dev server's PATH

You installed uv after the dev server was already running, so the server can't see it on PATH. Stop the server (Ctrl + C), open a fresh terminal (so both npm and uv resolve), cd into hunting-agent/, and run npm run dev again from there.

Fix MCP-2 — -32000: Connection closed even though uv is installed

The threat-intel server crashed at startup; the real error is hidden behind this code

Surface the true error by building the server's environment manually. Stop the server, then from hunting-agent/ run:

# macOS / Linux
$ uv --directory ../mcp-security/server/gti sync
# Windows
> uv --directory ..\mcp-security\server\gti sync

This downloads Python + dependencies and prints the real error. Common reveals:

  • Linux Permission denied (os error 13) — your repo is under a system-owned path like /opt. Move it into your home folder and take ownership, then restart: 
    $ sudo mv /opt/antisyphon_course_26062026 "$HOME/"
$ sudo chown -R "$USER:$USER" "$HOME/antisyphon_course_26062026"
    Never run the project with sudo.
  • Windows Access is denied (os error 5) — antivirus/Defender locked the uv cache mid-build. Retry the sync; if it persists, run uv cache clean then retry; if still stuck, add a Defender folder exclusion for %LOCALAPPDATA%\uv. Don't mix Administrator and normal terminals.

Once uv sync finishes cleanly, restart npm run dev and retry Lab 05.

Fix MCP-3 — Lab 05 fails or hangs with no clear error

Prerequisite missing, or first run still fetching deps

Confirm Section 4: uv is installed and VT_APIKEY is set in .env, and that you started the server from a terminal opened after installing uv. The first run is slow while uv fetches dependencies — give it a moment. If it errors rather than just being slow, use Fix MCP-2 to reveal the real cause.

Fix MCP-4 — Windows: uv install "requires an execution policy…"

PowerShell's default Restricted policy blocks uv's installer

Run Set-ExecutionPolicy RemoteSigned -Scope CurrentUser (answer Y), then re-run the uv install line from Section 4. It only affects your user account and is the standard dev setting.

8.8 · Fixes — Networking & misc

Fix NET-1 — "Port 5173 is in use"

Not a problem

Vite automatically picks the next free port. Just open whatever URL it prints in the terminal.

Fix NET-2 — Headless VM / WSL2 / remote box can't be reached from the host browser

Vite binds to localhost only by default

Start the server so it listens on all interfaces:

$ npm run dev:host

dev:host is a ready-made script for vite dev --host (npm run dev -- --host does the same). Then your forwarded / LAN port can reach it. On an untrusted network, prefer SSH port-forwarding instead — ssh -L 5173:localhost:5173 user@box — which needs no --host and keeps the server bound to localhost.
